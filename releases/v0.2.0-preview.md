# v0.2.0-preview

## tl;dr for teammates

- New pinned image: `kexinlu/mesh-memory:server-0.2.0-preview`.
- **Behavior change:** `recall` now returns lightweight thumbnails by
  default (no `content` body). Pass `hydrate: true` for the old shape,
  or use the new `get_many` tool to fetch bodies by id.
- Two new MCP tools: `get_many` and `suggest`.
- No schema migrations, no data changes. Safe to upgrade in place.

## Upgrading

1. Update your local compose:

   ```
   # In your mesh-memory deploy directory:
   git pull        # or re-download docker-compose.yml
   docker compose pull
   docker compose up -d
   ```

   The shipped `docker-compose.yml` pins
   `kexinlu/mesh-memory:server-0.2.0-preview`; if you had it edited to
   another tag, bump it manually.

2. Update your MCP bridge config in `~/.claude.json` — change
   `kexinlu/mesh-memory:mcp-0.1.1-preview` →
   `kexinlu/mesh-memory:mcp-0.2.0-preview`, then restart Claude Code.

No provisioning / token changes needed.

## What's new

### `recall` — now triage-first

Before: every `recall` call returned full memory bodies (`content`),
which bloated agent context as your memory grows.

After: `recall` returns a lightweight thumbnail — title, summary, and
scoring signals (importance, trust, recency, access count) — without
`content`. Triage there, then hydrate the ids you actually want.

```
# Triage
recall(query: "retry policy", project: "gateway")
  → [ { id, title, summary, importance, trust, ... }, ... ]

# Hydrate the ones you care about
get_many(ids: [id1, id2])
  → [ { ...full Memory, including content... }, ... ]
```

If you want the old behavior in a single call:

```
recall(query: "...", hydrate: true)
```

### `get_many` — the dedicated id-lookup surface

When a brief, event, or tool result hands you a memory id, this is
the tool to call. Works for a single id too — pass `ids: [id]`.

Do **not** pass a UUID as `recall`'s `query`; recall is for free-text
and anchor / kind / project filters, not id lookup.

### `suggest` — find related memories

Given a seed memory id, returns four buckets of related memories:

- **sameScope** — memories sharing an anchor with the seed (ranked by
  overlap count).
- **referenced** — everything wired by the refs graph: what the seed
  references, what references it, and the supersedes chain in both
  directions.
- **sameShape** — same `kind` + `project`, ranked by importance × trust.
- **textSimilar** — FTS on title + summary, ranked by `ts_rank`.

Buckets are **not** deduplicated. A memory that matches multiple
signals appears in each matching bucket — that overlap is itself a
relevance signal.

Returns lightweight thumbnails; pair with `get_many` to hydrate ids
that matter.

## Deprecations / removals

None.

## Known limits / deferred

- `suggest` is Tier 1 only today (anchor / refs / kind / FTS). Tier 2
  (co-access through team pods) and Tier 3 (semantic embeddings) are
  on the roadmap, not shipping here.
- No "cold-start" `suggest` — you need an existing seed memory id.
  A draft-signal variant is planned for v1.1.
