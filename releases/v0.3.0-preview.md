# v0.3.0-preview

## tl;dr for Operators and Agents

- L3 is now a compact memory cell: short summary, no `content` body.
- Large supporting material goes into explicit memory details.
- Upgrade runs one new migration: `20260418000000_l3_cell_detail_hygiene`.
- New server image: `kexinlu/mesh-context-server:0.3.0-preview`.
- Matching MCP image: `kexinlu/mesh-memory:mcp-0.3.0-preview`.
- Always run the L3 audit dry-run before cleanup.

## Upgrade

1. Pull the new images/config for `v0.3.0-preview`.
2. Start the server so migrations apply:

   ```bash
   docker compose pull
   docker compose up -d
   ```

3. Confirm the target before cleanup:

   ```bash
   scripts/l3_hygiene_audit.py \
     --database-url "$DATABASE_URL" \
     --api-url "$MESH_API_URL" \
     --token "$MESH_API_TOKEN"
   ```

4. If dry-run output is correct and L3 bloat exists, apply:

   ```bash
   scripts/l3_hygiene_audit.py \
     --database-url "$DATABASE_URL" \
     --api-url "$MESH_API_URL" \
     --token "$MESH_API_TOKEN" \
     --apply
   ```

`--apply` creates pre/post backups, moves existing L3 `content` into
`memory_details(detail_kind = legacy_content)`, trims long summaries, and does
not delete memories.

## Agent Write Rules

For L3 writes:

- `summary` must be 500 characters or fewer.
- `content` must be absent or `null`.
- communication/session-only kinds are rejected.
- include a meaningful registered anchor such as `topic`, `role`, `scope`,
  `area`, `stack`, `path`, or `workstream`.
- direct L3 creation must include `allow_new_canonical`, `proposed_l3`, or
  `supersedes`.

Use `writeMemoryDetail`, `memoryDetails`, and `deleteMemoryDetail` for large
evidence, examples, runbook bodies, or incident narratives.

```sql
from pg_attrdef

where attrelid = 'memories'::regclass and attname = 'fts';
```

The latest migration should be `20260418000000`, and the FTS expression should
use `title` + `summary` only.
